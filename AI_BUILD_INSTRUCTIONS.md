## Coding Standards

### TypeScript Style

- ESM modules only (`"type": "module"` in package.json)
- Strict TypeScript (`strict: true` in tsconfig)
- Maximum line length: 120 characters
- Use meaningful variable names (not `a`, `b`, `x`)
- Use `const` by default, `let` only when necessary
- Prefer immutability (readonly, ReadonlyArray)
- Use async/await for async operations (no callbacks)
- Use `Result<T, E>` pattern for operations that can fail
- Explicit return types on all public functions
- No `any` type — use `unknown` with type guards

### Architecture

- **Pure functions for business logic**: No framework dependencies in core/
- **Dependency injection**: Constructor injection for all dependencies
- **Single source of truth**: Vault/ledger is the source of truth
- **Unidirectional data flow**: Transaction -> Pipeline -> Seal -> Notify
- **Offline-first**: System works without network. Network is enhancement only.

### Documentation

- Every exported function MUST have JSDoc
- Every complex algorithm MUST have inline comments
- Use `// CONSTITUTION:` for constitutional constraint references
- Use `// NOTE:` for architectural explanations
- Use `// PERFORMANCE:` for optimization notes
- Update `BUILD_STATUS.md` when you complete a feature
- Update `MASTER_TASK_LIST.md` when you complete a task
- Update `KNOWN_BUGS.md` when you fix a bug

### Testing

- Write tests BEFORE or WITH production code (TDD preferred)
- Minimum 75% coverage for `src/core/` and `src/pipeline/`
- Every bug fix MUST include a regression test
- Use Node.js built-in test runner (via `tsx`)
- Determinism tests MUST pass (same input = same output)
